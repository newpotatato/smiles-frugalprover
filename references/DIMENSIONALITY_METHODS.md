# Активации, intrinsic dimension и методы сжатия — рабочие заметки

Дополнение к `PIPELINE.md`. Разбирает конкретно узел Budget Oracle: что такое активации gℓ(x), зачем нужна оценка intrinsic dimension (ID), как работает TwoNN, в чём именно состоит гипотеза H1 (и где наивная формулировка ломается), и почему UMAP — не замена TwoNN.

---

## 1. Что такое активации

Активации — внутренние численные представления, которые модель строит на каждом слое при обработке входа. Для условия задачи x и слоя ℓ, gℓ(x) — это вектор (обычно от сотен до нескольких тысяч чисел на токен), снятый **одним forward pass'ом, без генерации ответа**. Это не ответ модели, а её "внутреннее состояние" в момент обработки условия — снимок того, что происходит внутри, пока модель ещё не начала решать задачу.

## 2. Это фиче-инжиниринг — но особого рода

В отличие от классического фиче-инжиниринга (φ(x): длина текста, наличие формул и т.п. — признаки, придуманные вручную), gℓ(x) — это признак, **извлечённый из уже готового представления**, которое модель сама выучила для другой цели (предсказания следующего токена). По сути — аналог использования эмбеддингов из предобученной сети как признаков для отдельной downstream-задачи.

Но работа с ним всё равно требует классических решений фиче-инжиниринга:
- выбор слоя ℓ,
- способ агрегации по токенам (среднее? последний токен?),
- сжатие размерности (тысячи чисел → компактный признак),
- обучение лёгкой головы fθ поверх результата.

## 3. Intrinsic Dimension (ID) — что это и зачем

ID — минимальное число независимых "степеней свободы", нужное для описания геометрии облака точек, даже если формально оно живёт в пространстве в тысячи измерений. Это **не** то же самое, что "размерность, максимально сохраняющая информацию" (как в information bottleneck или пороге объяснённой дисперсии PCA) — ID определяется геометрически, через то, как быстро растёт число соседей вокруг точки при увеличении радиуса.

**Почему не PCA.** PCA — линейный метод: ищет плоскость, наилучшим образом приближающую данные. Если многообразие изогнуто (например, скручено, как "swiss roll"), PCA либо переоценивает размерность, либо не видит структуру вовсе. У Ansuini et al. прямо показано: искомый эффект **не воспроизводится** линейной оценкой — нужен нелинейный метод.

**Классические нелинейные альтернативы** (для контекста, не используются в проекте напрямую):
- Correlation dimension (Grassberger–Procaccia) — по степенному росту числа пар точек в шаре радиуса r;
- MLE-эстиматор Левиной–Бикеля — по правдоподобию расстояний до k соседей.

Оба чувствительны к неоднородной плотности данных — путают "разреженность области" с "размерностью".

## 4. TwoNN — алгоритм (Facco, D'Errico, Rodriguez, Laio, 2017)

Ключевой трюк: использовать не абсолютные расстояния, а **отношение** расстояний до двух ближайших соседей — в этом отношении локальная плотность точек математически сокращается.

**Шаг 1.** Для каждой точки i найти расстояние до первого соседа r₁ и до второго соседа r₂ (нужно всего 2 соседа на точку).

**Шаг 2.** Посчитать μᵢ = r₂ / r₁.

**Шаг 3 (ключевой факт).** Если локально данные распределены как однородный пуассоновский процесс на d-мерном многообразии с некоторой локальной плотностью ρ, то μ = r₂/r₁ подчиняется распределению Парето с параметром d:

f(μ) = d · μ^(−d−1), μ ≥ 1

ρ в этой формуле **отсутствует** — плотность полностью сокращается при переходе к отношению. Отсюда устойчивость метода к неоднородной плотности и к кривизне: не нужно предполагать постоянство плотности по всему датасету, только локальную однородность в микроокрестности каждой точки.

**Шаг 4.** Из теоретической функции распределения следует: −log(1 − F(μ)) = d · log(μ). Строим эмпирическую F̂(μ) по отсортированным μᵢ, откладываем −log(1−F̂(μᵢ)) против log(μᵢ) — должна получиться прямая через начало координат с наклоном d. Оценка d = обычная линейная регрессия через начало координат.

**Почему устойчив при малом N и на кривых многообразиях:**
- используются только 2 соседа на точку → короткие расстояния локально почти линейны даже на изогнутой поверхности;
- отношение расстояний инвариантно к масштабу локальной плотности;
- финальная оценка — регрессия по всем точкам сразу, шум усредняется.

**Как это применяет Ansuini et al.** Не изобретают новый эстиматор — берут готовый TwoNN и применяют послойно: для каждого слоя ℓ считают ID по облаку активаций на многих входах. Получившаяся кривая "ID от глубины" — горбообразная (hunchback): растёт в первых слоях, падает к выходу. ID последнего скрытого слоя коррелирует с итоговой test-accuracy — но это сравнение **между разными обученными сетями** (архитектурами/режимами обучения), а не между разными входами внутри одной фиксированной сети.

## 5. Где наивная формулировка гипотезы ломается

Соблазнительная, но неточная формулировка: "ID финальных слоёв напрямую коррелирует с качеством предсказания для конкретного входа x". Проблема — **ID нельзя посчитать по одной точке**: TwoNN нужно облако точек (нужны соседи, чтобы получить r₁, r₂). Одиночный вектор gℓ(x) сам по себе не имеет "своей ID".

Отсюда — два разных, не эквивалентных способа задействовать ID в проекте:

**(а) Диагностический (прямая аналогия с Ansuini, соответствует G2).** Берём весь корпус задач, считаем облако gℓ по всем x, сравниваем геометрию подоблаков — например, ID "лёгких" задач против ID "трудных". Даёт характеристику корпуса в целом, не предсказание для новой задачи.

**(б) Предсказательный (то, что нужно самому oracle).** Нужен сигнал на уровне одной задачи. Здесь в fθ подаётся не число ID, а сырой вектор gℓ(x) (или локальная характеристика — например, плотность соседей вокруг x, тоже из r₁/r₂, но без агрегации в одно глобальное d). Проверяемое условие — I(gℓ; B*) > 0 (mutual information сверх того, что даёт φ(x)), а не корреляция с ID.

**Вывод:** TwoNN/ID в проекте — вероятнее инструмент для G2 (объяснить, почему предиктор работает, через геометрию кластеров сложности), а не признак, который напрямую подаётся в fθ для предсказания конкретного B̂(x).

## 6. UMAP — как работает и чем отличается от TwoNN

UMAP (McInnes, Healy, Melville, 2018) — не оценщик размерности, а алгоритм **проекции** в заранее заданную размерность.

**Шаг 1 — граф в исходном пространстве.**
1. Для каждой точки i находим k ближайших соседей (`n_neighbors`).
2. ρᵢ — расстояние до самого близкого соседа (гарантирует связность).
3. σᵢ — локальный масштаб, калибруемый так, чтобы сумма весов до k соседей была фиксированной (аналог perplexity в t-SNE).
4. Направленный вес: w(i→j) = exp(−(d(i,j) − ρᵢ) / σᵢ).
5. Симметризация через fuzzy-union: w(i,j) = w(i→j) + w(j→i) − w(i→j)·w(j→i).

Теоретическая база — риманова геометрия + fuzzy simplicial sets (алгебраическая топология): данные лежат на многообразии с локально меняющейся метрикой, граф — приближение его топологии, устойчивое к неравномерной плотности выборки.

**Шаг 2 — оптимизация низкоразмерного представления.**
1. Задаём целевую размерность `n_components` (обычно 2 для визуализации).
2. Инициализация — обычно через спектральное вложение (Laplacian Eigenmaps).
3. Вес в низкой размерности: w_low(i,j) = 1 / (1 + a·dist(i,j)^(2b)), a и b подбираются под `min_dist`.
4. Минимизация кросс-энтропии между весами high-dim и low-dim графов: притяжение для соседей, отталкивание для не-соседей.
5. Оптимизация — SGD с negative sampling (как в word2vec) вместо полного O(N²) перебора пар — источник скорости UMAP на больших N.

**Ключевые гиперпараметры:** `n_neighbors` (локальность vs глобальность структуры), `min_dist` (плотность упаковки в проекции), `n_components` (целевая размерность — **задаётся пользователем**).

## 7. TwoNN vs UMAP — не взаимозаменяемы

| | TwoNN | UMAP |
|---|---|---|
| Что делает | Оценивает d — минимальную размерность, нужную для описания данных | Проецирует данные в d измерений, которое вы указали заранее |
| Выход | Число (оценка ID) | Набор координат в низкой размерности |
| Если задать n_components=2, а реальная ID = 15 | Не применимо — сам метод и есть способ узнать реальную ID | Молча впишет данные в 2D с искажениями, без предупреждения |
| Роль в проекте | Формальная проверка H1 (G2) | Вспомогательное сжатие/визуализация |

**Где UMAP всё же полезен в проекте** — не вместо TwoNN, а рядом:
- визуальная диагностика для G2: спроецировать gℓ(x) в 2D и посмотреть, разделяются ли "лёгкие"/"трудные" задачи на кластеры — быстрая качественная проверка перед формальным TwoNN-анализом;
- компрессия признака для fθ: если сырой gℓ(x) слишком велик для лёгкой probe-регрессии, сжать его UMAP'ом (например, до 20-50 измерений) и уже на этом обучать fθ — альтернативный инженерный выбор признака, конкурент использованию сырых активаций напрямую.

Итого: TwoNN — про измерение истинной сложности данных, UMAP — про сжатие/визуализацию под заданную размерность. Для строгой проверки H1 нужен первый; второй может дополнять пайплайн, но не подменяет логику измерения.
